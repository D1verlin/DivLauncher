# Minecraft Launcher (Electron + React)

## Описание проекта
Пользовательский лаунчер для Minecraft, разработанный на базе Electron и React. Проект представляет собой готовый шаблон, который любой пользователь может адаптировать под свой проект. Лаунчер предоставляет интерфейс для выбора серверных сборок, автоматического скачивания обновлений файлов игры и управления процессом запуска клиента.

## Применяемые технологии
* **Среда выполнения:** Node.js, Electron
* **Интерфейс:** React, Framer Motion
* **Работа с сетью и файловой системой:** Axios, Adm-Zip, fs
* **Автоматические обновления:** electron-updater

## Требования к окружению
* Node.js (рекомендуется версия 16.x или выше)
* npm или yarn
* Аккаунт Cloudflare (или любого S3-совместимого хранилища) для размещения файлов

## Часть 1: Настройка удаленного хранилища (Cloudflare R2)

Для корректной работы системы автообновлений лаунчера и загрузки игровых сборок необходимо настроить бакет (bucket) в Cloudflare R2.

### 1.1. Создание бакета
1. Авторизуйтесь в панели управления Cloudflare и перейдите в раздел **R2 Object Storage**.
2. Нажмите **Create bucket**.
3. Введите уникальное имя бакета (например, `my-launcher-data`) и выберите подходящий регион.
4. Нажмите **Create bucket**.

### 1.2. Настройка публичного доступа (Public Access)
Лаунчеру требуются прямые HTTP/HTTPS ссылки для загрузки файлов без использования токенов авторизации.
1. Перейдите во вкладку **Settings** вашего бакета.
2. В разделе **Public Access** -> **Custom Domains** нажмите **Connect Domain**.
3. Введите ваш подготовленный поддомен (например, `files.yourdomain.com`) и подтвердите привязку. Cloudflare автоматически настроит DNS-записи и выпустит SSL-сертификат.
4. Убедитесь, что в настройках DNS для этого поддомена включено проксирование (оранжевое облако).

### 1.3. Настройка CORS
Обязательный шаг для работы прогресс-бара. Без этого конфигурация `axios` не сможет считывать размер файла (`Content-Length`).
1. Во вкладке **Settings** прокрутите вниз до раздела **CORS rules**.
2. Нажмите **Add CORS policy** и переключитесь в режим редактирования JSON.
3. Вставьте следующую конфигурацию и сохраните:
```json
[
  {
    "AllowedOrigins": ["*"],
    "AllowedMethods": ["GET", "HEAD"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["Content-Length", "Content-Type"],
    "MaxAgeSeconds": 3600
  }
]
```

### 1.4. Рекомендуемая структура файлов в R2
Создайте в корне бакета следующую структуру:
```text
/
├── updates/                         # Для обновлений самого лаунчера (.exe и latest.yml)
└── launcher-data/                   # Для конфигурации и файлов сборок
    ├── modpacks.json                # Главный конфигурационный файл
    ├── pack1/
    │   ├── assets/                  # Картинки и иконки
    │   └── pack1_v1.0.0.zip         # ZIP-архив с модами
    └── pack2/
        ├── assets/
        └── pack2_v1.0.0.zip
```

## Часть 2: Настройка локального проекта

### 2.1. Установка
1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/D1verlin/DivLauncher.git
   ```
2. Перейдите в директорию проекта:
   ```bash
   cd DivLauncher
   ```
3. Установите зависимости:
   ```bash
   npm install
   ```

### 2.2. Конфигурация лаунчера
Перед сборкой необходимо указать пути к вашему хранилищу R2.

1. **Настройка обновлений лаунчера:**
   Откройте файл `package.json`. Найдите блок `build` -> `publish` и укажите ссылку на папку `updates` в вашем бакете:
   ```json
   "publish": [
     {
       "provider": "generic",
       "url": "https://files.yourdomain.com/updates"
     }
   ]
   ```
2. **Настройка ссылки на список сборок:**
   Откройте файл `src/ui/App.jsx` и измените константу `MODPACKS_URL` на прямую ссылку к вашему `modpacks.json`:
   ```javascript
   const MODPACKS_URL = "https://files.yourdomain.com/launcher-data/modpacks.json";
   ```

### 2.3. Структура файла modpacks.json
Создайте файл `modpacks.json` в вашем хранилище R2. Он управляет доступными сборками. 

Пример минимальной конфигурации:
```json
[
  {
    "id": "my-custom-pack",
    "name": "Название Сборки",
    "bgImage": "https://files.yourdomain.com/launcher-data/pack1/assets/bg.jpg",
    "logo": "https://files.yourdomain.com/launcher-data/pack1/assets/logo.png",
    "icon": "https://files.yourdomain.com/launcher-data/pack1/assets/icon.png",
    "mcVersion": "1.20.1",
    "loaderType": "forge",
    "packVersion": "1.0.0",
    "packZipUrl": "https://files.yourdomain.com/launcher-data/pack1/pack1_v1.0.0.zip",
    "useZip": true,
    "clientDir": ".my-custom-pack-client"
  }
]
```
* **id**: Уникальный идентификатор сборки.
* **packVersion**: Версия архива (используется для проверки обновлений).
* **packZipUrl**: Прямая ссылка на скачивание архива с модами.
* **clientDir**: Название скрытой папки, которая будет создана в AppData для установки файлов.

## Часть 3: Запуск и Сборка

### 3.1. Режим разработки
Для тестирования работы интерфейса и логики:
```bash
npm start
```

### 3.2. Компиляция для релиза
Для создания исполняемого файла (.exe) и генерации манифеста обновлений:
```bash
npm run build
```
После завершения сборки перейдите в локальную папку `dist/`. Скопируйте созданный файл установщика (`.exe`) и файл `latest.yml` в ваш бакет R2 в папку `updates/`.

## Часть 4: Механизм обновлений

В проекте реализованы две независимые системы обновлений:

1. **Обновление лаунчера (Electron Updater):** При старте лаунчер сверяет свою версию с файлом `latest.yml` на сервере. При наличии новой версии загрузка происходит в фоновом режиме.
2. **Обновление игровых сборок (Моды/Конфиги):** Лаунчер сравнивает значение `packVersion` из серверного `modpacks.json` с версией, сохраненной локально в `localStorage`. 

**Правила обновления сборок:**
* Из-за кэширования файлов на CDN запрещено перезаписывать старый ZIP-архив новым файлом с таким же именем. 
* При выпуске обновления создайте новый архив с другим именем (например, `pack1_v1.0.1.zip`).
* Загрузите архив в R2, затем обновите поля `packZipUrl` и `packVersion` в `modpacks.json`. Лаунчер автоматически предложит пользователям загрузить обновление.
